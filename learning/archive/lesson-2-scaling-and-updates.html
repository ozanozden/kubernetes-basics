Lesson 4: Scaling & Rolling Updates

Goal: Experience Kubernetes orchestration - scale your app dynamically and perform zero-downtime updates.

📚 What You'll Learn

This lesson demonstrates Kubernetes' core value propositions:

  • Horizontal Scaling: Add/remove instances on demand
  • Rolling Updates: Deploy new versions without downtime
  • Rollback: Revert to previous versions instantly
  • Declarative Management: Describe desired state, K8s makes it happen

Part 1: Horizontal Scaling

What is Horizontal Scaling?

Horizontal scaling (scale out/in): Add or remove instances

Vertical scaling (scale up/down): Increase resources (CPU/RAM) per instance

Kubernetes excels at horizontal scaling because:

  • No downtime - new Pods start while old ones run
  • Elastic - scale based on load (manual or automatic)
  • Service automatically load balances across all replicas
Scale Out (2 → 5 replicas): Before: After: Pod #1 Pod #1 Pod #2 Pod #2 Pod #3 ← new Pod #4 ← new Pod #5 ← new Service load balances across all 5!

🔬 Experiment 1: Scale Up

Setup - Monitor Everything:

# Terminal 1: Watch Pods
kubectl get pods -w

# Terminal 2: Keep port-forward running
kubectl port-forward service/k8s-demo-service 8080:8080

Scale from 2 to 5 Replicas:

# Terminal 3
kubectl scale deployment k8s-demo --replicas=5

Watch Terminal 1:

  1. 3 new Pods appear with status: ContainerCreating
  2. They become Running one by one
  3. Total: 5 Pods running

Verify scaling:

kubectl get deployment k8s-demo
# READY should show 5/5

kubectl get pods | grep k8s-demo | wc -l
# Should return 5

Test Load Distribution:

# Terminal 4: Call endpoint repeatedly, watch Pod names in logs
for i in {1..10}; do
  curl -s http://localhost:8080/hello | jq -r '.timestamp'
  sleep 0.5
done

Traffic is distributed across all 5 Pods!

🔬 Experiment 2: Scale Down

kubectl scale deployment k8s-demo --replicas=1

Watch what happens:

  1. 4 Pods immediately show Terminating
  2. Kubernetes gracefully shuts them down
  3. 1 Pod remains Running
  4. Your app still works (test with curl!)

Scale back to 2 for the next part:

kubectl scale deployment k8s-demo --replicas=2

💡 What Just Happened?

Declarative approach:

# You declared: "I want 5 replicas"
kubectl scale deployment k8s-demo --replicas=5

# Kubernetes figured out:
# - Current state: 2 Pods running
# - Desired state: 5 Pods running
# - Action needed: Create 3 more Pods
# - Execution: Created 3 Pods automatically

Compare to doing this manually with Docker:

# You'd have to:
docker run ... k8s-demo:v1  # Start Pod 3
docker run ... k8s-demo:v1  # Start Pod 4
docker run ... k8s-demo:v1  # Start Pod 5
# Then manually configure load balancer
# Then monitor if any crash and restart them
# 😰 So much manual work!

Kubernetes does all of this automatically when you change one number: replicas.

Part 2: Rolling Updates

What is a Rolling Update?

Deploy a new version by gradually replacing old Pods with new ones.

Rolling Update Strategy (v1 → v2): Step 1: Start Step 2: +1 new Step 3: -1 old v1 Pod #1 v1 Pod #1 v2 Pod #3 ✨ v1 Pod #2 v1 Pod #2 v1 Pod #2 v2 Pod #3 ✨ (new) Step 4: +1 new Step 5: -1 old Step 6: Done v2 Pod #3 v2 Pod #3 v2 Pod #3 v1 Pod #2 v2 Pod #4 ✨ v2 Pod #4 v2 Pod #4 ✨ v1 Pod #2 At every step, Service has healthy Pods → Zero downtime!

Default behavior:

  • MaxSurge: 25% (can create extra Pods during update)
  • MaxUnavailable: 25% (how many can be down during update)
  • With 2 replicas: Creates 1 new, then terminates 1 old, repeats

🔬 Experiment 3: Rolling Update to v2

Step 1: Update Your Code

Open src/main/java/com/backend/k8sdemo/web/HelloController.java

Change the message:

// From:
Map.of("message", "Hello from K8s Demo!", ...)

// To:
Map.of("message", "Hello from K8s Demo v2!",
       "version", "2.0",
       "timestamp", ...)

Step 2: Build New Docker Image

# Build with v2 tag
docker build -t k8s-demo:v2 .

# Load into minikube
minikube image load k8s-demo:v2

# Verify it's loaded
minikube image ls | grep k8s-demo

Step 3: Monitor the Update

# Terminal 1: Watch Pods with more detail
kubectl get pods -w -o wide

# Terminal 2: Watch rollout status
kubectl rollout status deployment/k8s-demo

# Terminal 3: Keep port-forward running
kubectl port-forward service/k8s-demo-service 8080:8080

Step 4: Update Deployment

Update k8s/deployment.yaml - change image tag:

spec:
  template:
    spec:
      containers:
      - name: k8s-demo
        image: k8s-demo:v2  # Changed from v1 to v2

Apply the change:

kubectl apply -f k8s/deployment.yaml

Watch Terminal 1:

  1. New Pod appears: k8s-demo-xxxxx with ContainerCreating
  2. New Pod becomes Running
  3. Old Pod goes Terminating
  4. Process repeats for second Pod
  5. Update complete!

Watch Terminal 2:

Waiting for deployment "k8s-demo" rollout to finish: 1 out of 2 new replicas have been updated...
Waiting for deployment "k8s-demo" rollout to finish: 1 old replicas are pending termination...
deployment "k8s-demo" successfully rolled out

✓ Test During Update

While the update is happening, keep calling your endpoint:

# Terminal 4
while true; do
  curl -s http://localhost:8080/hello | jq '.message, .version'
  sleep 1
done

You'll see:

"Hello from K8s Demo!"
null  # v1 doesn't have version field

"Hello from K8s Demo v2!"  # New version!
"2.0"

"Hello from K8s Demo v2!"
"2.0"

Zero downtime! Traffic gradually shifts from v1 → v2 as Pods update.

💡 What Happened Under the Hood

When you ran kubectl apply -f k8s/deployment.yaml:

  1. Kubernetes detected change: Image tag changed from v1 to v2
  2. Created ReplicaSet: New ReplicaSet for v2 Pods
  3. Scaled up new: Created 1 Pod with v2 image
  4. Waited for ready: Ensured new Pod passes health checks
  5. Scaled down old: Terminated 1 v1 Pod
  6. Repeated: Until all Pods are v2

Check the ReplicaSets:

kubectl get replicasets

# You'll see two:
# k8s-demo-xxxxxxx   0   0   0   10m   # Old (v1) - scaled to 0
# k8s-demo-yyyyyyy   2   2   2   2m    # New (v2) - scaled to 2

Kubernetes keeps the old ReplicaSet for rollback!

Part 3: Rollback

What if v2 is Broken?

Kubernetes keeps rollout history. You can rollback to any previous version instantly.

🔬 Experiment 4: Rollback to v1

View Rollout History:

kubectl rollout history deployment/k8s-demo

# Output:
# REVISION  CHANGE-CAUSE
# 1         <none>
# 2         <none>

Revision 1 = v1, Revision 2 = v2

Rollback to Previous Version:

kubectl rollout undo deployment/k8s-demo

Watch the Pods:

  1. New Pods created with v1 image
  2. v2 Pods terminated
  3. Same rolling update process, but in reverse!

Test It:

curl http://localhost:8080/hello | jq

You're back to v1! No "version" field.

Rollback to Specific Revision:

# Go back to v2
kubectl rollout undo deployment/k8s-demo --to-revision=2

✓ Understanding Rollout Commands

Command What It Does
kubectl rollout status Watch current rollout progress
kubectl rollout history List all revisions
kubectl rollout undo Rollback to previous revision
kubectl rollout undo --to-revision=N Rollback to specific revision
kubectl rollout pause Pause an in-progress rollout
kubectl rollout resume Resume a paused rollout

Part 4: Update Strategies

RollingUpdate vs Recreate

Kubernetes supports two update strategies:

Strategy How It Works Downtime? Use Case
RollingUpdate
(default)		 Gradually replace old with new ❌ No Most apps, production
Recreate Kill all old, then start all new ✅ Yes Can't run v1 & v2 together
(e.g., database schema changes)

RollingUpdate Configuration:

spec:
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1        # Max extra Pods during update
      maxUnavailable: 0  # Min Pods that must stay running

maxSurge: 1, maxUnavailable: 0 means:

  • Can create 1 extra Pod temporarily (2 → 3 during update)
  • All original Pods must stay running until new ones are ready
  • Zero unavailability guaranteed!

🔬 Experiment 5: Zero-Downtime Update

Update Deployment with Strict Settings:

Add to k8s/deployment.yaml:

spec:
  replicas: 2
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 0
  # ... rest of spec

Apply it:

kubectl apply -f k8s/deployment.yaml

Create v3 with a Slow Startup:

Update HelloController.java:

@RestController
public class HelloController {

    public HelloController() throws InterruptedException {
        // Simulate slow startup (10 seconds)
        Thread.sleep(10000);
    }

    @GetMapping("/hello")
    public ResponseEntity<Map<String, String>> sayHiToTheWorld() {
        Map<String, String> response = Map.of(
            "message", "Hello from K8s Demo v3!",
            "version", "3.0",
            "timestamp", Instant.now().toString()
        );
        return ResponseEntity.ok(response);
    }
}

Build and load:

docker build -t k8s-demo:v3 .
minikube image load k8s-demo:v3

Update deployment.yaml to v3, apply it, and monitor:

# Terminal 1: Watch Pods
kubectl get pods -w

# Terminal 2: Continuous requests
while true; do
  curl -s http://localhost:8080/hello 2>&1 | grep -o '"version":"[^"]*"' || echo "Failed"
  sleep 0.5
done

What you'll see:

  1. New v3 Pod created
  2. Stays in ContainerCreating for ~10s (slow startup)
  3. Old v2 Pods keep serving traffic!
  4. Once v3 Pod is Running, one v2 Pod terminates
  5. Process repeats for second Pod
  6. Zero failed requests! (maybe 1-2 if timing is tight)

This is the power of maxUnavailable: 0 - old Pods stay up until new ones are ready.

🤔 Key Takeaways

Scaling:

  • Change replicas field → Kubernetes creates/destroys Pods
  • Service automatically load balances across all replicas
  • Can scale to 100+ instances (limited by cluster resources)

Rolling Updates:

  • Change image tag → Kubernetes gradually replaces Pods
  • Zero downtime with proper maxSurge/maxUnavailable settings
  • Keeps old ReplicaSets for rollback

Rollback:

  • Instant rollback to any previous revision
  • Same rolling process, but in reverse
  • Production safety net - bad deploy? Roll it back!

Real-World Impact:

  • Black Friday? Scale from 5 to 50 replicas in seconds
  • Bug in production? Rollback in 30 seconds
  • Deploy during business hours? Zero downtime, users don't notice

⚠️ Production Considerations

In real environments, you'd also configure:

  • Health checks: Readiness/Liveness probes (K8s doesn't route to unhealthy Pods)
  • Resource limits: CPU/memory requests and limits
  • Autoscaling: Horizontal Pod Autoscaler (HPA) scales based on CPU/metrics
  • Pod Disruption Budgets: Limit how many Pods can be down during maintenance
  • Graceful shutdown: Handle SIGTERM, drain connections before terminating

These are beyond the scope of this intro, but good to know they exist!
