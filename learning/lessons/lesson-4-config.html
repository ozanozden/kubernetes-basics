🔧 Lesson 4: ConfigMaps (Configuration Management)

What you'll learn:
  • How to externalize configuration from container images
  • How ConfigMaps inject environment variables into pods
  • How to change configuration without rebuilding images
  • When to use ConfigMaps vs hardcoded values
Time: ~15 minutes
🤔 The Problem

Configuration is hardcoded in your container image. To change a port, log level, or feature flag, you must:
  1. Edit application code/config files
  2. Rebuild the image
  3. Push to registry
  4. Update Kubernetes deployment
This is slow, error-prone, and couples configuration to code.

How do you change config without rebuilding the image?
💡 The Solution: ConfigMaps

A ConfigMap is a Kubernetes object that stores non-sensitive configuration as key-value pairs. You can inject these values into pods as:
  • Environment variables (simple key-value pairs)
  • Files (mounted as volumes)
Benefit: Change config by updating the ConfigMap, then restart pods. No image rebuild needed.

Step 1: Create a ConfigMap

Create k8s/configmap.yaml:

apiVersion: v1
kind: ConfigMap
metadata:
  name: k8s-demo-config
data:
  # Key-value pairs that will become environment variables
  APP_NAME: "K8s Demo Production"
  LOG_LEVEL: "INFO"
  FEATURE_FLAG_ASYNC: "true"
Field Breakdown:

Field What It Is Why We Need It
apiVersion: v1 ConfigMap API version Tells K8s this is a core v1 resource
kind: ConfigMap Resource type Identifies this as configuration data
metadata.name Unique name for this ConfigMap Referenced by pods to load config
data Key-value pairs (strings only) Your actual configuration values

Step 2: Add Code to Read Configuration

Update src/main/java/com/backend/k8sdemo/web/HelloController.java to read environment variables:

package com.backend.k8sdemo.web;

import org.springframework.beans.factory.annotation.Value;
import org.springframework.http.ResponseEntity;
import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.RestController;

import java.time.Instant;
import java.util.Map;

@RestController
public class HelloController {

    // Inject environment variables with defaults
    @Value("${APP_NAME:K8s Demo}")
    private String appName;

    @Value("${LOG_LEVEL:INFO}")
    private String logLevel;

    @Value("${FEATURE_FLAG_ASYNC:false}")
    private boolean asyncEnabled;

    @GetMapping("/hello")
    public ResponseEntity<Map<String, String>> sayHiToTheWorld(){
        Map<String, String> response = Map.of(
                "message", "Hello from Kubernetes!",
                "version", "1.0",
                "timestamp", Instant.now().toString()
        );
        return ResponseEntity.ok(response);
    }

    // NEW: Endpoint to show configuration
    @GetMapping("/config")
    public String config() {
        return String.format(
            "App: %s | Log Level: %s | Async: %s",
            appName, logLevel, asyncEnabled
        );
    }
}
What does @Value do?

  • @Value("${APP_NAME:K8s Demo}") reads the APP_NAME environment variable
  • The part after : is the default value if env var doesn't exist
  • Spring Boot injects these values when the app starts
This lets you change config without changing code!

Step 3: Test Locally with Environment Variables

Before deploying, test that the code reads environment variables correctly:

# Set environment variables
export APP_NAME="Local Test"
export LOG_LEVEL="DEBUG"
export FEATURE_FLAG_ASYNC="true"

# Run the app
./gradlew bootRun

# In another terminal:
curl http://localhost:8080/config

Should output: App: Local Test | Log Level: DEBUG | Async: true

Press Ctrl+C to stop the app.

Step 4: Rebuild Docker Image

The code has changed (added /config endpoint), so rebuild:

# 1. Clean and rebuild the JAR
./gradlew clean bootJar

# 2. Build new Docker image
docker build -t k8s-demo:latest .

# 3. Load into minikube
minikube image load k8s-demo:latest

Step 5: Inject ConfigMap into Deployment

Update k8s/deployment.yaml to use ConfigMap values as environment variables:

spec:
  containers:
  - name: k8s-demo
    image: k8s-demo:latest
    ports:
    - containerPort: 8080
    # Inject ConfigMap values as environment variables
    env:
    - name: APP_NAME
      valueFrom:
        configMapKeyRef:
          name: k8s-demo-config
          key: APP_NAME
    - name: LOG_LEVEL
      valueFrom:
        configMapKeyRef:
          name: k8s-demo-config
          key: LOG_LEVEL
    - name: FEATURE_FLAG_ASYNC
      valueFrom:
        configMapKeyRef:
          name: k8s-demo-config
          key: FEATURE_FLAG_ASYNC
    resources:
      requests:
        memory: "256Mi"
        cpu: "250m"
      limits:
        memory: "512Mi"
        cpu: "500m"
How Environment Variable Injection Works:

  • env - Array of environment variables to set in the container
  • name: APP_NAME - The env var name inside the container
  • valueFrom.configMapKeyRef - Pull value from a ConfigMap instead of hardcoding
  • name: k8s-demo-config - Which ConfigMap to read from
  • key: APP_NAME - Which key in that ConfigMap
When the pod starts, Kubernetes reads the ConfigMap and sets APP_NAME environment variable to "K8s Demo Production".

Step 6: Deploy to Kubernetes

# Apply ConfigMap and Deployment
kubectl apply -f k8s/configmap.yaml
kubectl apply -f k8s/deployment.yaml

# Force restart to use new image
kubectl rollout restart deployment/k8s-demo
kubectl rollout status deployment/k8s-demo
🧪 Experiment: Change Config Without Rebuilding

  1. Test the /config endpoint: 
    kubectl port-forward service/k8s-demo-service 8080:8080
curl http://localhost:8080/config
    Should show: App: K8s Demo Production | Log Level: INFO | Async: true
  2. Update ConfigMap (no rebuild needed!): 
    kubectl edit configmap k8s-demo-config
# Change LOG_LEVEL: "INFO" to LOG_LEVEL: "DEBUG"
# Save and exit (:wq in vim)
  3. Restart pods to pick up new config: 
    kubectl rollout restart deployment/k8s-demo
kubectl rollout status deployment/k8s-demo
  4. Verify change: 
    curl http://localhost:8080/config
    Now shows: App: K8s Demo Production | Log Level: DEBUG | Async: true
  5. Check ConfigMap contents: 
    kubectl get configmap k8s-demo-config -o yaml

ConfigMap Best Practices

✅ Use ConfigMaps For:
  • Environment-specific settings (dev/staging/prod)
  • Feature flags
  • Log levels
  • Service URLs (non-sensitive)
  • Application ports
❌ Don't Use ConfigMaps For:
  • Passwords, API keys, tokens (use Secrets instead - Lesson 5)
  • Large files (>1MB - use volumes instead)
  • Binary data (base64 bloats YAML)

Alternative: envFrom (Inject All Keys)

Instead of mapping each key individually, inject all ConfigMap keys at once:

spec:
  containers:
  - name: k8s-demo
    image: k8s-demo:latest
    # Inject ALL keys from ConfigMap as env vars
    envFrom:
    - configMapRef:
        name: k8s-demo-config

This creates environment variables for every key in the ConfigMap automatically.
