⚙️ Lesson 3: Resource Management (CPU & Memory Limits)

What you'll learn:
  • How to prevent resource starvation in Kubernetes
  • The difference between requests and limits
  • What happens when a pod exceeds memory limits (OOMKilled)
  • How Kubernetes schedules pods based on resources
Time: ~10 minutes
🤔 The Problem

A pod with a memory leak can consume all available RAM on the node, causing:
  • Other pods to be evicted (killed)
  • Node crashes
  • Cascading failures across your cluster
Similarly, a CPU-hungry pod can starve other workloads.

How do you prevent one misbehaving pod from taking down the cluster?
💡 The Solution: Resource Requests & Limits

Kubernetes lets you specify two resource constraints:

Type Purpose What Happens
Requests Minimum resources guaranteed K8s won't schedule pod on a node without this much available
Limits Maximum resources allowed Pod is throttled (CPU) or killed (memory) if exceeded

Example: If you request 256Mi memory, K8s guarantees your pod gets at least that much. If you limit to 512Mi, the pod is killed if it tries to use more.

Step 1: Add Resource Constraints

Update k8s/deployment.yaml to add resource requests and limits:

spec:
  containers:
  - name: k8s-demo
    image: k8s-demo:latest
    ports:
    - containerPort: 8080
    # Resource requests and limits
    resources:
      requests:
        memory: "256Mi"     # Minimum memory needed
        cpu: "250m"         # 0.25 CPU cores minimum
      limits:
        memory: "512Mi"     # Maximum memory allowed
        cpu: "500m"         # 0.5 CPU cores maximum
    livenessProbe:
      httpGet:
        path: /actuator/health/liveness
        port: 8080
      initialDelaySeconds: 30
      periodSeconds: 10
    readinessProbe:
      httpGet:
        path: /actuator/health/readiness
        port: 8080
      initialDelaySeconds: 10
      periodSeconds: 5
Understanding CPU & Memory Units:

CPU:
  • 1 = 1 full CPU core
  • 500m = 0.5 cores (m = millicores, 1000m = 1 core)
  • 250m = 0.25 cores
Memory:
  • Mi = Mebibytes (1024-based: 256Mi = 268,435,456 bytes)
  • M = Megabytes (1000-based: 256M = 256,000,000 bytes)
  • Common: 128Mi, 256Mi, 512Mi, 1Gi (1 Gibibyte)
Why requests < limits?
Allows bursting. Your app normally uses 256Mi but can temporarily spike to 512Mi. Requests ensure scheduling; limits prevent runaway resource usage.
⚠️ What Happens When Limits Are Exceeded?

CPU: Pod is throttled (slowed down but not killed)
Memory: Pod is OOMKilled (Out Of Memory killed) and restarted

You'll see this in pod events: Reason: OOMKilled

Step 2: Apply and Verify

kubectl apply -f k8s/deployment.yaml
kubectl rollout status deployment/k8s-demo
🧪 Experiment: See Resource Limits in Action

  1. Check resource allocation: 
    kubectl describe pod <pod-name> | grep -A 5 "Limits\|Requests"
    You'll see the configured requests and limits for each container.
  2. Check scheduling decisions: 
    kubectl get pod <pod-name> -o yaml | grep -A 10 resources
  3. Watch pod events for resource issues: 
    kubectl get events --sort-by='.lastTimestamp' | grep -i oom
    (You won't see OOMKilled unless a pod exceeds memory limits)

How Kubernetes Uses Requests & Limits

Scheduling (Requests):
When you create a pod requesting 256Mi memory, Kubernetes finds a node with at least 256Mi available. If no node has enough, the pod stays Pending.

Enforcement (Limits):
Once running, if the pod tries to use more than 512Mi, the kernel kills it (OOMKilled). Kubernetes then restarts it (if liveness probe is configured).

Why both?
- Requests = "What I need to run"
- Limits = "What I'm allowed to spike to"
- Without limits: one pod can hog all resources
- Without requests: Kubernetes might schedule too many pods on one node

Best Practices

✅ Good Resource Configuration:
  • Set both requests and limits - Prevents scheduling issues and resource starvation
  • Requests = typical usage - Based on monitoring/profiling
  • Limits = 2x requests - Allows bursting without being wasteful
  • Start conservative - You can always increase later
❌ Common Mistakes:
  • No limits - One pod can crash the node
  • Limits too low - Constant OOMKills during normal operation
  • Requests too high - Pods stay Pending, wasting cluster capacity
  • Requests = Limits - No bursting allowed (OK for some workloads, restrictive for others)

Summary

✅ What you learned:
  • Requests guarantee minimum resources for scheduling
  • Limits prevent runaway resource consumption
  • CPU limits throttle pods, memory limits kill pods (OOMKilled)
  • Resource management is critical for cluster stability
Key Commands:

kubectl get events --sort-by='.lastTimestamp'
kubectl describe node
