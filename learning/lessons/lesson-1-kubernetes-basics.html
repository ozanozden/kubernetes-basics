Lesson 3: Deploy to Kubernetes (Explained)

Goal: Understand what each Kubernetes resource does and why we need it, then write the manifests yourself.

📚 The Problem We're Solving

With Docker alone, you manually:

  • Run containers: docker run
  • Restart crashed containers manually
  • Scale by running more containers manually
  • Handle load balancing yourself
  • Update by stopping old, starting new containers

Kubernetes automates all of this. But to tell K8s what to do, we write manifests - YAML files that describe our desired state.

Understanding Kubernetes Resources

🎯 The Three Resources We'll Create

Resource What It Does Why We Need It
Pod Wraps your container(s) K8s doesn't run containers directly - it runs Pods
Deployment Manages multiple identical Pods Handles scaling, updates, and keeping desired number of Pods running
Service Stable network endpoint Load balances traffic across Pods (Pod IPs change, Service IP doesn't)

Part 1: Understanding Deployments

What is a Deployment?

A Deployment is a controller that manages Pods. Think of it as a manager that:

  • Creates the number of Pods you specify (replicas)
  • Monitors them - if one crashes, creates a replacement
  • Handles updates - gradually replaces old Pods with new ones
  • Handles rollbacks if updates fail

You don't create Pods manually. You create a Deployment, and it creates/manages Pods for you.

Building deployment.yaml Line by Line

Let's build this file understanding each piece:

apiVersion: apps/v1
What it does: Tells K8s which API group this resource belongs to.
Why: K8s has multiple API versions for different resource types. Deployments are in the apps/v1 API.
Think of it as: Import statement - "I'm using the apps API, version 1"
kind: Deployment
What it does: Declares what type of resource this is.
Why: Tells K8s "this YAML describes a Deployment, not a Service or Pod"
Options: Deployment, Service, Pod, ConfigMap, Secret, etc.
metadata:
  name: k8s-demo
What it does: Names this Deployment.
Why: You'll reference this name in kubectl commands: kubectl get deployment k8s-demo
Must be unique: Within a namespace, no two Deployments can have the same name.
spec:
  replicas: 2
What it does: Tells K8s to run 2 copies (Pods) of your application.
Why:
  • High availability: If one Pod crashes, the other keeps serving traffic
  • Load distribution: Traffic is split between both Pods
  • Zero downtime updates: Can update one while the other serves traffic
You can change this: Set to 1 for testing, 3+ for production
  selector:
    matchLabels:
      app: k8s-demo
What it does: Tells the Deployment how to find "its" Pods.
Why: In K8s, multiple Deployments/Pods exist. This selector says "I manage Pods with label app=k8s-demo"
Must match: The labels in template.metadata.labels below
Think of it as: SQL WHERE clause - "SELECT pods WHERE app='k8s-demo'"
  template:
    metadata:
      labels:
        app: k8s-demo
What it does: Template for Pods this Deployment creates. Labels each Pod with app=k8s-demo
Why:
  • Matches the selector above (Deployment finds these Pods)
  • Service uses this label to find Pods to send traffic to
  • You can query: kubectl get pods -l app=k8s-demo
Labels are key-value pairs - you can have multiple: app: k8s-demo, version: v1, tier: backend
    spec:
      containers:
      - name: k8s-demo
What it does: Pod spec - describes what containers run in each Pod. This names the container.
Why: A Pod can run multiple containers (sidecar pattern). Each needs a unique name.
We only have one container - our Spring Boot app
        image: k8s-demo:latest
What it does: Specifies which Docker image to run.
Why: This is your containerized app! K8s pulls this image and runs it.
Format: repository:tag or registry/repository:tag
Our case: Local image (no registry), tag v1
        imagePullPolicy: Never
What it does: Tells K8s NOT to pull from a remote registry.
Why: Our image is local (loaded into minikube). Default behavior would try to pull from Docker Hub and fail.
Options:
  • Always - Always pull from registry (production)
  • IfNotPresent - Pull only if not cached locally
  • Never - Use local image only (our case)
        ports:
        - containerPort: 8080
What it does: Documents which port the container listens on.
Why:
  • Documentation - tells readers/tools "this app serves on 8080"
  • Used by Services to route traffic to the right port
  • Some networking plugins require this
Doesn't expose the port - that's the Service's job. This just declares it.

Complete deployment.yaml:

apiVersion: apps/v1
kind: Deployment
metadata:
  name: k8s-demo
spec:
  replicas: 2
  selector:
    matchLabels:
      app: k8s-demo
  template:
    metadata:
      labels:
        app: k8s-demo
    spec:
      containers:
      - name: k8s-demo
        image: k8s-demo:latest
        imagePullPolicy: Never
        ports:
        - containerPort: 8080

Part 2: Understanding Services

What is a Service?

Pods have dynamic IPs - they change when Pods restart/scale. A Service provides a stable endpoint.

The problem without Services:

Pod #1: 172.17.0.5 ← crashes Pod #2: 172.17.0.6 Pod #3: 172.17.0.9 ← new Pod, different IP! How do clients know which IPs to call?

The solution with Services:

Service: stable IP (10.96.0.10) + DNS name ↓ Load balances to: Pod #1: 172.17.0.5 Pod #2: 172.17.0.6 Pod #3: 172.17.0.9 Clients always call the Service IP/DNS!

Building service.yaml Line by Line

apiVersion: v1
What it does: Services are in the core K8s API (v1), not apps/v1
Why different from Deployment: Services are older/more fundamental than Deployments
kind: Service
What it does: Declares this is a Service resource
metadata:
  name: k8s-demo-service
What it does: Names this Service
Why: This becomes the DNS name! Other Pods can call http://k8s-demo-service:8080
Used in commands: kubectl get service k8s-demo-service
spec:
  type: NodePort
What it does: Exposes the Service on each Node's IP at a static port
Why we need it: For local testing with minikube. Lets us access from outside the cluster.
Service types:
Type Use Case
ClusterIP Internal only (default) - not accessible from outside cluster
NodePort Exposes on Node IP (minikube, testing)
LoadBalancer Cloud provider creates external load balancer (AWS ELB, GCP LB)
  selector:
    app: k8s-demo
What it does: Tells Service which Pods to send traffic to
Why: Matches the labels on Pods created by your Deployment!
How it works: Service queries "find all Pods with label app=k8s-demo" and routes traffic to them
Must match: The labels in Deployment's template.metadata.labels
  ports:
  - protocol: TCP
What it does: Specifies the network protocol
Why TCP: HTTP runs over TCP. Most services use TCP.
Other option: UDP (for DNS, video streaming, etc.)
    port: 8080
What it does: The port the Service listens on (what clients connect to)
Why: Clients call k8s-demo-service:8080
Can be different from targetPort - e.g., Service on 80, Pods on 8080
    targetPort: 8080
What it does: The port on the Pods that the Service forwards traffic to
Why: Must match containerPort in Deployment (where your app listens)
Traffic flow: Client → Service:8080 → Pod:8080 → Your app

Complete service.yaml:

apiVersion: v1
kind: Service
metadata:
  name: k8s-demo-service
spec:
  type: NodePort
  selector:
    app: k8s-demo
  ports:
  - protocol: TCP
    port: 8080
    targetPort: 8080

How They Work Together

1. You create Deployment ↓ 2. Deployment creates 2 Pods (replicas: 2) - Each Pod labeled with app=k8s-demo - Each Pod runs your k8s-demo:latest container - Each container listens on port 8080 ↓ 3. You create Service ↓ 4. Service finds Pods with label app=k8s-demo - Creates stable endpoint (IP + DNS) - Load balances traffic between the 2 Pods ↓ 5. You access: http://<minikube-ip>:<nodeport>/hello ↓ 6. Service routes to → Pod #1 or Pod #2 (round-robin) ↓ 7. Your Spring Boot app responds

Now Write the Files Yourself

Create k8s/deployment.yaml and k8s/service.yaml based on the explanations above.

Don't copy-paste! Type them out. If you understand each line, you'll remember it.

Tips:

  • YAML is indentation-sensitive (use spaces, not tabs)

  • Lists use - prefix
  • Check your indentation carefully!

✓ Deploy and Test

Once you've written both files:

# Apply them
kubectl apply -f k8s/deployment.yaml
kubectl apply -f k8s/service.yaml

# Check status
kubectl get deployments
kubectl get pods
kubectl get services

# Test - port forward in one terminal
kubectl port-forward service/k8s-demo-service 8080:8080

# Then in another terminal, test the endpoint
curl http://localhost:8080/hello

🔬 Experiment: Kubernetes Self-Healing

Now see Kubernetes' killer feature in action!

Setup - Watch Pods in Real-Time:

# Terminal 1: Watch pods continuously
kubectl get pods -w

Leave this running. You'll see pod status changes instantly.

Setup - Keep Traffic Flowing:

# Terminal 2: Port-forward your service
kubectl port-forward service/k8s-demo-service 8080:8080

Setup - Monitor Responses:

# Terminal 3: Call endpoint repeatedly
while true; do curl -s http://localhost:8080/hello | jq '.timestamp'; sleep 1; done

You'll see timestamps updating every second. Note which Pod names appear in timestamps.

The Experiment - Kill a Pod:

# Terminal 4: Delete one Pod
kubectl get pods  # Copy a pod name
kubectl delete pod <pod-name>

Watch what happens:

  1. Terminal 1 (pod watch): Pod goes Terminating → New Pod appears → Running
  2. Terminal 3 (curl loop): Requests keep working! Maybe one fails, but service continues
  3. Deployment: Immediately creates a replacement Pod to maintain replicas: 2
Before delete: Pod #1 (k8s-demo-abc123) ← DELETE THIS Pod #2 (k8s-demo-xyz789) ← Serves traffic During delete: Pod #1 (k8s-demo-abc123) → Terminating... Pod #2 (k8s-demo-xyz789) ← All traffic goes here Pod #3 (k8s-demo-new456) → Creating... After delete: Pod #2 (k8s-demo-xyz789) ← Serving Pod #3 (k8s-demo-new456) ← Serving (new one!) Deployment says: "I need 2 replicas. Only 1 running? CREATE NEW POD!"

This is self-healing! Kubernetes constantly monitors desired state (2 replicas) vs actual state (1 running) and fixes the gap automatically.

Try This Too:

# Scale down to 1 replica
kubectl scale deployment k8s-demo --replicas=1

# Watch: One Pod terminates
# Only 1 Pod remains running

# Scale back up to 2
kubectl scale deployment k8s-demo --replicas=2

# Watch: New Pod appears!

Stop the curl loop: Press Ctrl+C in Terminal 3
