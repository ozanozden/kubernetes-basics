🔒 Lesson 5: Secrets (Sensitive Data)

What you'll learn:
  • How to handle sensitive data in Kubernetes
  • The difference between Secrets and ConfigMaps
  • How base64 encoding works (and why it's NOT encryption)
  • Production secret management best practices
Time: ~15 minutes
🤔 The Problem

ConfigMaps are great for non-sensitive config, but what about:
  • Database passwords
  • API keys
  • OAuth tokens
  • TLS certificates
ConfigMaps are stored in plain text and visible to anyone with cluster access.

How do you handle sensitive data securely?
💡 The Solution: Secrets

A Secret is like a ConfigMap but designed for sensitive data:
  • Values are base64-encoded (not encrypted, but obscured)
  • Can be encrypted at rest (cluster-level configuration)
  • Access can be restricted via RBAC (role-based access control)
  • Not shown in logs or kubectl describe output
Note: Base64 is NOT encryption. For production, enable encryption at rest in your cluster.

Step 1: Create a Secret

Create k8s/secret.yaml:

apiVersion: v1
kind: Secret
metadata:
  name: k8s-demo-secret
type: Opaque
data:
  # Values must be base64-encoded
  DB_PASSWORD: cGFzc3dvcmQxMjM=     # "password123" in base64
  API_KEY: YWJjZGVmMTIzNDU2         # "abcdef123456" in base64
⚠️ How to Base64 Encode:

# Encode
echo -n "password123" | base64
# Output: cGFzc3dvcmQxMjM=

# Decode
echo "cGFzc3dvcmQxMjM=" | base64 -d
# Output: password123
NEVER commit real secrets to git! This is just for demonstration.
Secret Types:

Type Use Case
Opaque Generic key-value secrets (most common)
kubernetes.io/tls TLS certificates (requires tls.crt and tls.key)
kubernetes.io/dockerconfigjson Docker registry credentials
kubernetes.io/basic-auth Basic authentication (username/password)

Step 2: Add Code to Use Secrets

Update src/main/java/com/backend/k8sdemo/web/HelloController.java to add secret handling:

package com.backend.k8sdemo.web;

import org.springframework.beans.factory.annotation.Value;
import org.springframework.http.ResponseEntity;
import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.RestController;

import java.time.Instant;
import java.util.Map;

@RestController
public class HelloController {

    // ConfigMap values
    @Value("${APP_NAME:K8s Demo}")
    private String appName;

    @Value("${LOG_LEVEL:INFO}")
    private String logLevel;

    @Value("${FEATURE_FLAG_ASYNC:false}")
    private boolean asyncEnabled;

    // NEW: Secret values
    @Value("${DB_PASSWORD:default-password}")
    private String dbPassword;

    @Value("${API_KEY:default-key}")
    private String apiKey;

    @GetMapping("/hello")
    public ResponseEntity<Map<String, String>> sayHiToTheWorld(){
        Map<String, String> response = Map.of(
                "message", "Hello from Kubernetes!",
                "version", "1.0",
                "timestamp", Instant.now().toString()
        );
        return ResponseEntity.ok(response);
    }

    @GetMapping("/config")
    public String config() {
        return String.format(
            "App: %s | Log Level: %s | Async: %s",
            appName, logLevel, asyncEnabled
        );
    }

    // NEW: Endpoint to show masked secrets
    @GetMapping("/secrets")
    public String secrets() {
        // NEVER log actual secrets in production!
        return String.format(
            "DB Password: %s | API Key: %s",
            maskSecret(dbPassword),
            maskSecret(apiKey)
        );
    }

    private String maskSecret(String secret) {
        if (secret.length() <= 4) return "****";
        return secret.substring(0, 2) + "****" + secret.substring(secret.length() - 2);
    }
}
Why mask secrets?

Never return raw secrets in API responses or logs! The maskSecret method shows only the first/last 2 characters so you can verify the secret is loaded without exposing it.

Step 5: Inject Secrets into Deployment

Update k8s/deployment.yaml to use Secret values:

spec:
  containers:
  - name: k8s-demo
    image: k8s-demo:latest
    ports:
    - containerPort: 8080
    env:
    # ConfigMap values
    - name: APP_NAME
      valueFrom:
        configMapKeyRef:
          name: k8s-demo-config
          key: APP_NAME
    - name: LOG_LEVEL
      valueFrom:
        configMapKeyRef:
          name: k8s-demo-config
          key: LOG_LEVEL
    # Secret values (same pattern, different source)
    - name: DB_PASSWORD
      valueFrom:
        secretKeyRef:
          name: k8s-demo-secret
          key: DB_PASSWORD
    - name: API_KEY
      valueFrom:
        secretKeyRef:
          name: k8s-demo-secret
          key: API_KEY
Comparison:

ConfigMap: configMapKeyRef
Secret: secretKeyRef

The injection pattern is identical - only the source type changes!

Step 6: Deploy to Kubernetes

# Apply Secret and Deployment
kubectl apply -f k8s/secret.yaml
kubectl apply -f k8s/deployment.yaml

# Force restart to use new image
kubectl rollout restart deployment/k8s-demo
kubectl rollout status deployment/k8s-demo
🧪 Experiment: Verify Secrets Work

  1. Test secrets endpoint: 
    kubectl port-forward service/k8s-demo-service 8080:8080
curl http://localhost:8080/secrets
    Should show masked values: DB Password: pa****23 | API Key: ab****56
  2. Verify secrets are NOT visible in describe: 
    kubectl describe pod <pod-name> | grep -i password
    You won't see the actual password.
  3. See secret metadata (not values): 
    kubectl get secret k8s-demo-secret -o yaml
    Values are base64-encoded, not plain text.
  4. Decode a secret manually: 
    kubectl get secret k8s-demo-secret -o jsonpath='{.data.DB_PASSWORD}' | base64 -d
    This proves base64 is NOT encryption - anyone with kubectl access can decode!

🔒 CRITICAL: Production Secret Management

The Problem: Base64 is NOT encryption - anyone can decode it. Committing secret.yaml to git exposes your secrets!

Production Solutions:
  • Don't commit secrets to git - Add k8s/secret.yaml to .gitignore
  • Encrypted Secrets: Use Sealed Secrets or SOPS to encrypt before committing
  • External Vaults: Use HashiCorp Vault, AWS Secrets Manager, Azure Key Vault, or Google Secret Manager
  • CI/CD Secrets: Store in GitHub Secrets, GitLab CI Variables, etc. and inject during deployment
For this lesson: We're using demo values that are safe to commit. Real projects need one of the above solutions.

Sealed Secrets (Example Production Solution)

How Sealed Secrets Work:

1. Install Sealed Secrets controller in your cluster
2. Encrypt your secret with kubeseal CLI
3. Commit the encrypted SealedSecret to git
4. Controller decrypts it into a regular Secret at runtime

Example: 
# Create sealed secret (can be committed!)
kubeseal < secret.yaml > sealed-secret.yaml

# Apply sealed secret
kubectl apply -f sealed-secret.yaml

# Controller creates the actual Secret automatically
Benefit: Your secret is in git, encrypted. Only the cluster can decrypt it.

Summary

✅ What you learned:
  • Secrets are like ConfigMaps but for sensitive data
  • Base64 encoding obscures but does NOT encrypt
  • Production secrets require encryption (Sealed Secrets, Vault, etc.)
  • Secret injection uses secretKeyRef instead of configMapKeyRef
Key Commands: 
kubectl get secret k8s-demo-secret -o yaml
kubectl get secret k8s-demo-secret -o jsonpath='{.data.DB_PASSWORD}' | base64 -d
kubectl describe pod <name> | grep -i secret

Production Checklist

Before Going to Production:
  • ✅ Enable encryption at rest in your cluster
  • ✅ Use Sealed Secrets or external vault (Vault, AWS Secrets Manager)
  • ✅ Never commit secret.yaml with real credentials to git
  • ✅ Restrict Secret access with RBAC
  • ✅ Rotate secrets regularly
  • ✅ Audit secret access logs
