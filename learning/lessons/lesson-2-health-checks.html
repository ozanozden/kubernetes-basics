🏥 Lesson 2: Health Checks (Liveness & Readiness Probes)

What you'll learn:
  • How Kubernetes knows if your app is alive vs ready
  • The difference between liveness and readiness probes
  • How to configure health checks in Kubernetes
  • Why health probes prevent broken pods from serving traffic
Time: ~15 minutes
🤔 The Problem

Imagine your app crashes internally - maybe an OutOfMemoryError or a deadlock. The container is still running, so Kubernetes thinks everything is fine. Traffic keeps flowing to a broken pod.

Or consider startup: your app takes 30 seconds to boot. Kubernetes starts sending traffic immediately, and requests fail for 30 seconds.

How does Kubernetes know your app is actually healthy?
💡 The Solution: Health Probes

Kubernetes offers two types of health checks:

Probe Type Purpose What Happens on Failure
Liveness Probe Is the app alive (not crashed/deadlocked)? Pod is killed and restarted
Readiness Probe Is the app ready to serve traffic? Pod is removed from service (no traffic) but not killed

Key Difference:
  • Liveness = "Are you dead?" (restart if broken)
  • Readiness = "Are you ready?" (wait if not ready)

Step 1: Add Spring Boot Actuator Dependency

Spring Boot Actuator provides production-ready health endpoints out of the box. First, add it to build.gradle:

dependencies {
    implementation 'org.springframework.boot:spring-boot-starter-web'
    // Add Spring Boot Actuator for health endpoints
    implementation 'org.springframework.boot:spring-boot-starter-actuator'
    testImplementation 'org.springframework.boot:spring-boot-starter-test'
}
What is Spring Boot Actuator?

Actuator adds production-ready features to your Spring Boot app:
  • Health checks - /actuator/health endpoints
  • Metrics - Memory usage, request counts, etc.
  • Info - Application metadata
For Kubernetes, we only need the health check endpoints.

Step 2: Configure Health Endpoints

Create src/main/resources/application.yaml with this configuration:

management:
  endpoints:
    web:
      exposure:
        include: health,info
  endpoint:
    health:
      probes:
        enabled: true
  health:
    livenessState:
      enabled: true
    readinessState:
      enabled: true
What do these properties do?
  • management.endpoints.web.exposure.include - Which actuator endpoints to expose over HTTP
  • management.endpoint.health.probes.enabled=true - Enable Kubernetes-style liveness/readiness probes
  • management.health.livenessState.enabled=true - Expose /actuator/health/liveness
  • management.health.readinessState.enabled=true - Expose /actuator/health/readiness
These endpoints return HTTP 200 (OK) or 503 (Service Unavailable) based on app state.

Step 3: Test Locally First

Before deploying to Kubernetes, verify the health endpoints work locally:

# Start the app
./gradlew bootRun

# In another terminal, test the endpoints:
curl http://localhost:8080/actuator/health
curl http://localhost:8080/actuator/health/liveness
curl http://localhost:8080/actuator/health/readiness

You should see JSON responses like:

{
  "status": "UP",
  "groups": ["liveness", "readiness"]
}

Press Ctrl+C to stop the app once verified.

⚠️ Important: Don't Skip Local Testing!

Always test code changes locally before deploying to Kubernetes. This catches issues early and saves time debugging pod crashes.

Step 4: Rebuild Docker Image

Now that the code has changed (added actuator), rebuild the Docker image:

# 1. Clean and rebuild the JAR
./gradlew clean bootJar

# 2. Build new Docker image
docker build -t k8s-demo:latest .

# 3. Load into minikube
minikube image load k8s-demo:latest
Why these steps?
  • ./gradlew clean bootJar - Compiles Java code and packages it into a JAR file
  • docker build - Creates a new Docker image with the new JAR
  • minikube image load - Copies the image into minikube's local registry
Without these steps, Kubernetes will use the old image without actuator endpoints!

Step 5: Update Deployment with Health Probes

Now update k8s/deployment.yaml to tell Kubernetes about the health endpoints:

apiVersion: apps/v1
kind: Deployment
metadata:
  name: k8s-demo
spec:
  replicas: 3
  selector:
    matchLabels:
      app: k8s-demo
  template:
    metadata:
      labels:
        app: k8s-demo
    spec:
      containers:
      - name: k8s-demo
        image: k8s-demo:latest
        ports:
        - containerPort: 8080
        # NEW: Liveness probe - restart if unhealthy
        livenessProbe:
          httpGet:
            path: /actuator/health/liveness
            port: 8080
          initialDelaySeconds: 30    # Wait 30s after container starts
          periodSeconds: 10            # Check every 10 seconds
          timeoutSeconds: 5            # Probe must respond within 5s
          failureThreshold: 3          # Kill pod after 3 consecutive failures
        # NEW: Readiness probe - remove from service if not ready
        readinessProbe:
          httpGet:
            path: /actuator/health/readiness
            port: 8080
          initialDelaySeconds: 10    # Check readiness sooner than liveness
          periodSeconds: 5             # Check more frequently (every 5s)
          timeoutSeconds: 3            # Faster timeout
          failureThreshold: 3          # Remove from service after 3 failures
Understanding the Fields:

Field What It Means Why We Need It
httpGet.path HTTP endpoint to call Tells K8s where to check health
httpGet.port Port to probe Must match your app's port
initialDelaySeconds Wait time before first probe Gives app time to start up (readiness < liveness)
periodSeconds How often to probe Readiness should check more frequently
timeoutSeconds How long to wait for response Prevent hanging on slow responses
failureThreshold Failures before taking action Avoid false positives from transient issues

Step 6: Deploy to Kubernetes

# Apply the updated deployment
kubectl apply -f k8s/deployment.yaml

# Force Kubernetes to use the new image
kubectl rollout restart deployment/k8s-demo

# Watch the rollout
kubectl rollout status deployment/k8s-demo
Why kubectl rollout restart?

Even though we loaded a new image, Kubernetes won't automatically pull it because the tag (:latest) hasn't changed. The restart command forces Kubernetes to recreate pods with the new image.

Alternative: Use version tags (:v1, :v2) and update the deployment YAML. We use :latest + restart for simplicity during learning.

Watch the rolling update respect readiness probes - pods won't receive traffic until ready!

🧪 Experiment: See Health Probes in Action

  1. Watch pod readiness during rollout: 
    kubectl get pods -w
    Notice pods show 0/1 READY initially, then 1/1 READY after readiness probe passes.
  2. Check probe configuration: 
    kubectl describe pod <pod-name> | grep -A 10 "Liveness\|Readiness"
    You'll see the configured probe settings and their status.
  3. Delete a pod and watch self-healing: 
    kubectl delete pod <pod-name>
kubectl get pods -w
    Kubernetes immediately creates a new pod. Readiness probe prevents traffic until it's ready!
  4. Check events to see probe activity: 
    kubectl get events --sort-by='.lastTimestamp' | grep -i probe

Why Liveness vs Readiness?

Real-World Scenarios:

Liveness Probe Use Case:
Your app deadlocks. It's running but not responding. Liveness probe fails → Kubernetes restarts the pod → Problem solved.

Readiness Probe Use Case:
Your app is connecting to a database during startup. It's alive but not ready to serve requests. Readiness probe fails → Kubernetes doesn't send traffic → Once DB connection succeeds, readiness passes → Traffic starts flowing.

Why both?
Readiness is temporary (waiting for resources). Liveness is permanent (app is broken and needs restart). Using both gives Kubernetes the full picture of your app's health.

Common Mistakes

⚠️ Avoid These Pitfalls:
  • initialDelaySeconds too short: Probe fails during startup → pod killed before it finishes booting → crash loop
  • Same probe for liveness and readiness: Temporary issues (like DB connection loss) trigger liveness → pod restart → makes problem worse
  • No timeout: Slow probes can delay pod restarts or traffic routing
  • failureThreshold = 1: One transient blip kills your pod → unstable deployment

Summary

✅ What you learned:
  • Liveness probes detect broken pods and restart them automatically
  • Readiness probes prevent traffic to pods that aren't ready yet
  • Probe timing (initialDelay, period, timeout, failureThreshold) fine-tunes behavior
  • Self-healing - Kubernetes keeps your app healthy without manual intervention
Key Commands: 
kubectl describe pod <name> | grep -A 10 Liveness
kubectl get events --sort-by='.lastTimestamp'
kubectl get pods -w
