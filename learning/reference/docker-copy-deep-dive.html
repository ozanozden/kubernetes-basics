Lesson 2b: Docker COPY Deep Dive

Goal: Understand exactly how COPY works, why order matters, and how .dockerignore filters files.

📚 The Build Context

When you run docker build ., the . is your build context - the directory Docker sends to the daemon.

What happens:

  1. Docker packages your entire project directory (excluding .dockerignore entries)
  2. Sends this package to the Docker daemon
  3. The daemon can now access these files for COPY/ADD instructions
Your machine: Docker daemon: /k8s-demo/ Build context (temp): ├── src/ ├── src/ ├── build.gradle ├── build.gradle ├── Dockerfile ├── Dockerfile ├── .git/ ─────────→ (not sent - in .dockerignore) ├── .gradle/ ─────────→ (not sent - in .dockerignore) └── learning/ ─────────→ (not sent - in .dockerignore)

Why this matters: Large build contexts = slow builds. If you have a 2GB .git directory, Docker sends all 2GB even if you never COPY it!

Understanding COPY Syntax

Basic Format

COPY <src>... <dest>

Rules:

  • <src> is relative to the build context (not your Dockerfile location)
  • <dest> is an absolute path in the container, OR relative to WORKDIR
  • If <dest> ends with /, it's treated as a directory
  • If <dest> doesn't exist, Docker creates it

Example 1: Copy Single File

WORKDIR /app
COPY build.gradle .

Result: Copies build.gradle from build context to /app/build.gradle in container

Explanation:

  • build.gradle = source file in your project root
  • . = current WORKDIR (/app)
  • Destination becomes /app/build.gradle

Example 2: Copy Multiple Files

WORKDIR /app
COPY build.gradle settings.gradle gradlew ./

Result: Copies all three files to /app/

Rules for multiple sources:

  • Last argument MUST be a directory (end with / or be .)
  • All previous arguments are source files/directories
  • Files keep their names in destination

Example 3: Copy Directory

WORKDIR /app
COPY src ./src

Result: Copies src/ directory to /app/src/

What gets copied:

Build context: Container: src/ /app/src/ ├── main/ ├── main/ │ └── java/ │ └── java/ └── test/ └── test/

Example 4: Copy Everything

WORKDIR /app
COPY . .

Result: Copies entire build context to /app/

What this means:

  • First . = source (build context root)
  • Second . = destination (WORKDIR = /app)
  • Copies ALL files/directories from build context
  • Respects .dockerignore - excluded files won't be copied
Build context (after .dockerignore): Container: /k8s-demo/ /app/ ├── src/ ├── src/ ├── build.gradle ├── build.gradle ├── gradle/ ├── gradle/ ├── Dockerfile ├── Dockerfile └── gradlew └── gradlew NOT copied (.dockerignore): ├── .git/ ├── .gradle/ └── learning/

Understanding .dockerignore

What is .dockerignore?

A file that tells Docker which files/directories to exclude from the build context. Works like .gitignore.

Your .dockerignore:

.git
.gradle
build
.idea
*.md
learning/

Pattern Matching:

Pattern Matches Example
.git Exact directory name .git/
*.md All .md files (any depth) README.md, docs/guide.md
learning/ Directory and contents learning/lessons/
**/*.log All .log files at any depth app.log, logs/error.log
!important.md Exception (include this file) Overrides *.md pattern

With vs Without .dockerignore

❌ Without .dockerignore:

$ docker build .
Sending build context to Docker daemon  245.7MB  ← SLOW!

Includes: .git/ (200MB), .gradle/ (30MB), learning/ (5MB), build/ (10MB)

✅ With .dockerignore:

$ docker build .
Sending build context to Docker daemon  156kB    ← FAST!

Only includes: src/, gradle/, build files, Dockerfile

Speed improvement: 245MB → 156KB (1500x faster!)

Why COPY Order Matters (Layer Caching)

Docker Layer Caching

Each Dockerfile instruction creates a layer. Docker caches layers and reuses them if nothing changed.

Cache invalidation rule: When a layer changes, all subsequent layers are invalidated.

❌ Bad Order (cache-unfriendly)

FROM gradle:8-jdk21 AS build
WORKDIR /app

# ❌ Copy everything at once
COPY . .

# Gradle downloads dependencies (slow)
RUN ./gradlew bootJar --no-daemon

Problem: Every code change invalidates COPY layer → Gradle re-downloads all dependencies

Build time after changing one line of code: ~2-3 minutes

✅ Good Order (cache-friendly)

FROM gradle:8-jdk21 AS build
WORKDIR /app

# ✅ Copy dependency files first
COPY build.gradle settings.gradle ./
COPY gradle ./gradle
COPY gradlew ./

# Download dependencies (cached if build.gradle unchanged)
RUN ./gradlew dependencies --no-daemon

# ✅ Copy source code last (changes frequently)
COPY src ./src

# Build (cached if src unchanged)
RUN ./gradlew bootJar --no-daemon

Benefit: Changing Java code doesn't invalidate dependency download layer

Build time after changing one line of code: ~30 seconds

Our Simplified Approach

COPY . .

Why we used this:

  • Simpler for learning (fewer lines)
  • .dockerignore protects us from copying unwanted files
  • For this learning project, build time isn't critical

Trade-off: Simplicity vs cache optimization. For production, use the multi-step COPY pattern above.

COPY vs ADD

There's also an ADD instruction. Here's when to use which:

Feature COPY ADD
Copy local files ✅ Yes ✅ Yes
Copy from URL ❌ No ✅ Yes
Auto-extract .tar files ❌ No ✅ Yes
Recommended ✅ Default choice ⚠️ Only when needed

Best practice: Use COPY unless you specifically need ADD's features. COPY is more transparent.

Multi-stage COPY

COPY --from=stage

In multi-stage builds, you can copy from earlier stages:

FROM gradle:8-jdk21 AS build
WORKDIR /app
COPY . .
RUN ./gradlew bootJar --no-daemon

FROM eclipse-temurin:21-jre AS runtime
WORKDIR /app
# Copy from the "build" stage
COPY --from=build /app/build/libs/*.jar /app/app.jar
ENTRYPOINT ["java", "-jar", "/app/app.jar"]

What happens:

  1. Build stage creates JAR in /app/build/libs/
  2. Runtime stage copies ONLY the JAR from build stage
  3. Build tools (Gradle, source code) stay in build stage
  4. Final image contains only JRE + JAR (much smaller!)

Size comparison:

  • Build stage image: ~800MB (Gradle + JDK + source)
  • Final runtime image: ~320MB (JRE + JAR only)

⚠️ Common COPY Mistakes

1. Wrong source path

❌ COPY /src ./src       # Absolute path - looks in filesystem root!
✅ COPY src ./src        # Relative to build context

2. Forgetting trailing slash

❌ COPY src /app/src     # Creates /app/src as a file if src doesn't exist!
✅ COPY src /app/src/    # Always creates as directory

3. Wildcard confusion

❌ COPY *.jar /app/      # Only works at build context root
✅ COPY build/libs/*.jar /app/

4. Not using .dockerignore

❌ COPY . .              # Without .dockerignore = copies .git, node_modules, etc.
✅ Create .dockerignore first!

🤔 Key Takeaways

  • Build context is the directory sent to Docker daemon
  • .dockerignore filters build context before sending (faster builds)
  • COPY . . copies entire build context (minus .dockerignore entries) to WORKDIR
  • Layer caching is why copy order matters (copy rarely-changed files first)
  • Multi-stage COPY lets you copy artifacts between build stages
  • Always use COPY unless you specifically need ADD's features
